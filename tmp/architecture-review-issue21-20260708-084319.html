Problem: the pending→active rule is expressed four times — an activate_product handler that applies it, a docstring that claims update_product applies it, a @model_validator that's a no-op, and a DB CHECK constraint that's the only thing actually enforcing it. A PATCH that sets price without flipping status passes validation and 500s at the DB.

Solution: one module owns the transition — given current status/price and a patch, it returns the next status/price or raises a domain error. update_product and activate_product both call it instead of each re-deriving the rule.

  • Locality: the rule lives in one place, not four
  • Interface enforces what the docstring only claimed
  • Bad PATCH becomes a 400, not a 500
  • Leverage: same predicate settles the pending-check duplication in checkout.py, products.py, CheckoutPage.tsx

Delete the phantom idempotency cache

Strong in-process

app/api/products.py:165-300

sequenceDiagram participant FE as frontend (fresh uid() every submit) participant Cache as _quick_add_idem_cache participant DB as unique constraint FE->>Cache: key = qa-r-barcode-RANDOM Cache-->>FE: miss (always — key is never repeated) FE->>DB: insert DB-->>FE: 409 on real double-tap (constraint, not cache)

Problem: the module exists to dedupe a same-key retry, but both frontend callers regenerate a random idempotency key on every submit — so the cache-hit branch is structurally unreachable through the actual UI, and no test sends the same key twice. Deletion test: remove lines 165–300, nothing observably changes.

Solution: delete the cache; the DB unique-constraint + 409 path already handles the real double-tap correctly. If key-stable retry is wanted, mirror CheckoutPage.tsx's idempotencyKeyRef pattern (a stable ref reset only on success) instead of a server-side cache.

  • Removes an abstraction that fails the deletion test
  • One fewer untested branch
  • If kept: fix at the seam that actually varies — the frontend key, not the server cache

Stop reconstructing pending-product origin at read time

Worth exploring in-process

app/api/products.py:442-557 (list_pending_products)

Before

flowchart LR L[list_pending_products] --> SL["scan ALL StockinLog\nrows"] L --> IL["scan ALL InvoicingLog\nrows"] SL --> J["python-side join\npick latest per product"] IL --> J classDef leak stroke:#dc2626,stroke-width:2px; class SL,IL leak

After

flowchart LR W["quick_add_product\nwrite time"] --> O["Product.origin\n(written once)"] L[list_pending_products] --> O classDef deep fill:#0f172a,color:#fff,stroke:#0f172a; class O deep

Problem: the origin (receiving vs checkout) is decided once, at write time, via the X-Quick-Add-Origin header — then has to be un-made at read time by scanning both log tables and picking the latest match. The handler's own comment flags this as a scaling risk deferred, not solved. Deletion test: removing the scan doesn't remove the need to know origin — it reappears as "which log table has this product" scattered wherever origin is needed next.

Solution: record origin directly on the row (or a dedicated column) at the same write that already knows it, instead of re-deriving it from two audit tables on every list call.

  • Locality: origin decided once, read once
  • Removes an unindexed full-table scan on a read path

Split the picker/phone login request shapes

Speculative in-process

app/schemas/auth.py · app/api/auth.py:126-174

Problem: LoginRequest carries staff_id and phone as optional siblings; "pick exactly one" is enforced imperatively inside login_shop, not by the schema. The frontend already fully committed to the picker (no leftover phone UI) — only the backend schema still models both forms as one type.

Solution: two request shapes routed to the shared _authenticate helper, or a discriminated union, so "exactly one identifier" is a type-level fact.

  • Interface states the invariant instead of hiding it in a branch

Low urgency: _authenticate itself is already a clean, reused seam — this only tidies the request shape above it.

Top recommendation

Centralize the pending → active transition

Of the six candidates, this is the only one where the current shallowness has an observable failure mode today: update_product can produce a 500 instead of a 400 because the transition rule lives in a docstring and a no-op validator instead of one interface. It also directly resolves the five-way duplicated pending/active status check (finding 5), so fixing it pays down two friction points at once. Start here — jump to card.
