Problem: five unrelated concerns (CRUD, quick-add, pending listing with a hand-rolled log join, activation, CSV import with its own validation state machine) sit directly in route handlers. Unlike checkout/voids/eod, there's no app/services/products.py — the router's interface is barely thinner than its implementation.

Solution: a ProductsService module behind the router, matching the pattern already used for checkout/voids/eod — one interface per concern, tested directly instead of only through the HTTP seam.

  • Interface finally thinner than implementation
  • Leverage: matches the existing services pattern elsewhere in the repo
  • Locality: CSV-import bugs and quick-add bugs stop sharing one 740-line blast radius

Encode the rollback-before-translate contract once

Strong in-process

app/db.py:83-96 · app/api/checkout.py · voids.py · lots.py · dashboard.py · products.py

flowchart LR G["get_db()\n(no auto session.begin)"] --> R1[checkout.py] G --> R2[voids.py] G --> R3[lots.py] G --> R4[dashboard.py] G --> R5[products.py] R1 -.own rollback+translate.-> DB1[(db)] R2 -.own rollback+translate.-> DB1 R3 -.own rollback+translate.-> DB1 R4 -.own rollback+translate.-> DB1 R5 -.own rollback+translate.-> DB1 classDef leak stroke:#dc2626,stroke-width:2px; class R1,R2,R3,R4,R5 leak

Problem: get_db deliberately skips wrapping sessions in session.begin(), so every mutating route is individually responsible for commit timing and calling rollback() before translating a domain error to HTTP. Five routers currently do this correctly, by hand, with nothing enforcing it — and no test specifically exercises "does a failed mutation leave the session usable for the next request."

Solution: a small context manager or decorator (e.g. with unit_of_work(db):) that owns commit-on-success/rollback-on-domain-error, used by all five routers instead of each re-deriving the convention.

  • Locality: one place a future route can't get wrong
  • Turns an implicit convention into an enforced interface

  • Leverage: one auth-header/error-parsing path, all four call sites

Fold the router error-logging convenience into _errors.py

Worth exploring in-process

app/api/checkout.py:55-82 · dashboard.py:42-54 · voids.py:67-86 · app/api/_errors.py

Problem: three routers each define their own _CODE_TO_STATUS dict and a private _error_to_http that calls map_error_to_http then logs at ERROR if the result is 500. Only the dict contents vary; the wrapping behavior is identical boilerplate three times.

Solution: map_error_to_http takes the code-map as a parameter and owns the "log unmapped codes" behavior itself, so routers stop re-writing the wrapper.

  • Leverage: one wrapper, three routers

Problem: three separate conventions are each reimplemented at N call sites: the "superadmin hasn't picked a shop" bail-out (4 pages), the "append shop_id to query/body" plumbing (6 api/*.ts modules), and the "turn a thrown error into a user message" catch block (4 pages). Each is small individually, but together they're the same shape — a cross-cutting concern solved locally instead of once.

Solution: a useShopScope()-backed guard component for the first, a withShopId() helper in client.ts for the second, and a toUserMessage(e, fallback) helper for the third.

  • Locality: three rules, three homes, not twelve
  • New pages get the guard/message-handling for free

Top recommendation

Split products.py into a Products module

It's the widest blast radius: five unrelated concerns share one 740-line router, it's the only major router without a services module counterpart, and two other findings here (the shop-scope read checks, the audit-log write shape) live partly inside it. Pulling it into a ProductsService — matching the pattern already proven by checkout/voids/eod — gives the router an interface thinner than its implementation and unblocks the smaller fixes around it. Jump to card.

Two testing gaps worth flagging separately (not deepening candidates, but risk): invoice_pdf.py's GSTIN/excise-duty placeholder rendering has no dedicated assertions beyond content-type/status, and no test simulates two concurrent checkout finalizes racing the last unit of stock — _current_stock_for's concurrency-safety argument (checkout.py:159-172) currently rests on a comment, not an executable check.
