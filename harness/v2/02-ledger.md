## From Phase 2 grill (2026-07-07)

D-v2-4 | Quick-add captures exactly two fields: **brand + size_label**. No price, no threshold, no free-text note. | Brand+size+optional note | User chose the plain two-field option — fastest path for low-literacy staff (matches v1 D-38/R-23), owner fills everything else.
D-v2-5 | Modeled as **the same `Product` table**, with a new `status` (`pending`/`active`) column; `price` becomes nullable only while `status = pending`. A pending Product is a real row everywhere (lots, stock views, catalog) — just not sellable and not price-set. | Separate `pending_products` staging table promoted into `Product` later | User chose same-table + status flag. Avoids duplicating lot/stock-join logic for a second table; only checkout needs a new "reject if pending" check (existing R-9 invariant: "every invoice line resolves to a real, in-stock product" already implies price-less products can't be finalized — this extends naturally rather than requiring new invariant logic).
D-v2-6 | A pending product's received quantity **counts as real stock immediately** — Lot creation against a pending product works exactly like against an active one (received_by, quantity, stockin_logs entry), per existing `lots.py` flow. Only `checkout` refuses to sell a pending product. | Stock held/staged until owner activates the product | User: "Yes, stock counts immediately." Matches the excise stock-register expectation (v1 R-34) that received units are accounted for the moment they're on the shelf, independent of pricing status.
D-v2-7 | At checkout, scanning a barcode that is either wholly unknown or already `pending` triggers the same quick-add flow (brand+size) if unknown, then shows a **blocking-but-scoped message**: "Pending — no price yet, contact admin," and that line is not addable to the cart. The rest of the sale is unaffected — cashier can continue scanning other items and finalize without the pending line. | Whole-transaction block until the pending barcode is manually removed | User chose the scoped-block option — keeps checkout throughput high (v1 D-4/D-39's 2000 invoices/day target) instead of stalling an entire sale over one bad scan.
D-v2-8 | The admin-facing surface is a single **"Pending Products" list/screen** (who added it, when, from receiving or checkout) with a dashboard badge/count. Completing the product (setting price, flipping `status` to `active`) *is* the resolution — there is no separate "acknowledge/dismiss" action or notifications table. | A distinct dismissible notifications record independent of product completion | User chose the simpler unified option. One less data model (no `notifications` table), and matches v1's existing owner-dashboard pattern (D-33/R-26 low-stock list) of "the list itself is the notification."

## Decisions made autonomously (mechanical, reusing already-established v1 patterns — not re-litigated with the user per this skill's dependency-closure rule)

D-v2-9 | Duplicate-barcode race on quick-add (two people quick-adding the same unknown code near-simultaneously) is caught by the **existing global unique constraint on `Product.barcode`** (v1 D-52) — the second write gets a DB conflict; UI shows "Someone already added this — refreshing" and re-resolves the barcode instead of retrying the create. | New app-level locking/de-dup logic | Reuses D-52 verbatim; this is a restatement of an existing invariant, not a new decision surface.
D-v2-10 | `owner` can also use quicksearch and the quick-add flow (not receiver/cashier-only), since `owner` is already a superset of both roles (v1 D-26). | Receiver/cashier-only quick-add UI | Direct consequence of D-26; not a new choice.
D-v2-11 | Quicksearch-by-name is implemented as a **client-side filter over the already-prefetched catalog** (`catalog.ts`'s `prefetchCatalog`, D-30) — add a parallel array/index for name search alongside the existing barcode-keyed `Map`, no new backend endpoint. | New server-side `/products/search` endpoint | Codebase check confirmed the catalog is already fully cached client-side (up to 500 active products) purely for fast barcode resolution; extending the same in-memory structure to support name substring search is a pure frontend change with no new network path, consistent with D-30's "no per-scan round trip" goal.
D-v2-12 | Quick-add POST reuses the same **idempotency-key pattern** as checkout finalize (D-30) so a double-tap "Add" doesn't attempt two creates for one barcode (the DB unique constraint, D-v2-9, is the ultimate backstop; the idempotency key avoids a needless error toast on the common double-tap case). | No idempotency key, rely solely on the DB constraint | Consistent with existing checkout-finalize pattern; avoids a jarring "already exists" error for an ordinary accidental double-submit.
D-v2-13 | Provisional-product creation is logged via the **existing domain-specific log tables** (D-47) — a `product.pending_created` event on `stockin_logs` when it happens during receiving, or on `invoicing_logs` when it happens during checkout — not a new log table. | New dedicated `product_logs` table | Reuses v1's established domain-log pattern (event_type string + payload, seen live in `lots.py`'s `StockinLog`); no new observability surface needed for one new event type.
D-v2-14 | Quick-add at checkout only ever creates the `Product` row (0 stock, since no Lot was recorded) — even after the owner sets its price and activates it, the item still can't be sold until an actual Lot is received for it, per the existing "stock derived from lots, never negative" invariant (v1 R-9/D-17). No special-case bypass is added for checkout-originated pending products. | Allowing a checkout-created pending product to be sold once priced, without ever being received into a Lot | Falls directly out of already-frozen v1 invariants — flagging here so it's explicit rather than silently "just working," since a checkout-time quick-add without a matching lot receipt would otherwise look like an unexplained stock gap in the audit trail.
